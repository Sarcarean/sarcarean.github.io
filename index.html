 My Thoughts...(Regarding "Big Ball of Mud" and the Blog post)
I am all too familiar with the aforementioned “Big Ball of Mud” structure. Having worked on code basis with hundreds or thousands of source files (lines count in the millions), I have seen enterprises do it right and many that do it wrong. However, the fabled mud is less an issue (and obsolete) these days, as while the languages have evolved, so too has the IDE’s. We are not living in the days of using primitive development environments, but modern, controlled, and organized environments that create seamless structures that automatically untangle, refract, and rebuild. I no longer have to think of living in an IDE world without CodeLens, Intellisense, change history, automatic definition renaming, code base synchronizing and so on and so on.

 DevOps summary
DevOps is a series of software development methodologies. Its goal is to automate much of the traditional software development cycles and make it easy (and faster) to delivery large scale projects to customers via the internet (or cloud as its known). DevOps uses several strategic designs to implement a better pipeline for software with a focus to improve deployment, faster time to market, better security, lower bug rates and when there are bugs, faster response time to fix this.

 Inevitable Pain summary

    This academic paper discusses the methods and thought about creating meaningful, bug free (or bug minimum) software. The scope of the discussion covers a very large range, as it starts about a discussion on how to create an early match making algorithm (circa 1960s) and goes into more modern programming (as it covers UNIX and GUI development). Several examples of programming models are discussed, including build-and-fix, the waterfall model, structured programming, requirements engineering and extreme programming.

    The paper concludes that there is no single best solution for software development, while some methods may work best in some scenarios it may not in others. Programming is an art, and each person is an artist that uses different methods to solve and tackle different problems. I can relate this article to the programming I do professionally. Often, I will rewrite entire libraries that someone else has written, not because it was bad or flawed, but because my interpretation may work best for me in the run long as I expand or maintain it for future versions.

 Software is hard!

    This article makes the argument that software projects (generally defined as multi-person teams) are complex and that is it difficult to measure progress. It explores one common metric: line counts and correctly states that it is a bad metric for which to measure scale. The article gives examples of why it takes longer to program one program versus another and that some projects might take multitudes longer to develop. I can relate, I have written 100 lines of VHDL that is far more complex than 1000 lines of C/C++. I would conclude by saying software is hardware but it gets easier.

Description: this project demonostrates how to properly interface with a 1.8V high density serial memory device in BGA-24 package. I designed and built the hardware (4 layer PCB), and programmed the firmware and software in order to accomplish this. For the peripheral controller, I used a 32-bit ARM processor (Cortex-M3) with 480MHz PLL with a high-speed complex programmable device (CPLD). The USB 2.0 high-speed stack is implemented and is capable of transfering 34MB/s between the PC and peripheral memory. If you want to see the complete design, take a look at the Hardware schematic.



Description: This is a demonstration of a typical development trifecta, that is a project that uses high-level desktop software (.net), firmware (C/C++), and programmable logic (VHDL). I just made this short video for my client (Cypress Semiconductor) that I was contracted to implement a hardware programming solution for their top of the line memory product capable of 333MB/s data transfer. The protocol is comprised of a 8-bit parallel IO, clocked with a differential signal with 50% duty cycle and data being loaded at the rising and falling edges of the clock.
